# Roadmap

Built one milestone at a time. Each milestone is only started after the previous one is confirmed working.

- [x] **Milestone 0 — Architecture & repository setup**
  Vision alignment, tech stack decisions, dataset selection, repo skeleton, docs (this folder).

- [x] **Milestone 1 — Data acquisition & exploration**
  Pull raw data samples from Chicago, NYC, and SF open data APIs; verify live schemas against `docs/DATA_SOURCES.md`; exploratory notebooks; write a data dictionary for each city's raw schema. See `notebooks/01-03_*.ipynb` and `docs/DATA_DICTIONARY.md`. Key findings: Chicago has a native `domestic` flag, NYC has per-complaint demographic fields (governed by `docs/ETHICS.md`) with ~33-38% unusable/unknown race values, SF has the cleanest native neighborhood field. Cross-city "domestic-related" comparisons can't be measured identically across all three cities — a limitation to state explicitly in the site's narrative.

- [x] **Milestone 2 — Data pipeline & cross-city normalization**
  Full pipeline built in `scripts/` (fetch -> clean/normalize -> geospatial join -> aggregate), documented in `docs/PIPELINE.md`. Pulled a real 2-year window: 1,652,117 incidents across all three cities, geo-joined to 380 neighborhoods (100% match for Chicago/NYC, 95.3% for SF), normalized into a shared 21-category taxonomy, and reduced to a 98,759-row / 264KB monthly aggregate table that Milestone 3's database will actually serve.

- [x] **Milestone 3 — Database & backend API**
  PostgreSQL schema (Alembic migrations), seed script loading the pipeline's processed parquet files, FastAPI app with typed endpoints (`/api/cities`, `/api/categories`, `/api/neighborhoods`, `/api/aggregates`, `/api/summary`), 13 passing tests. Revised mid-build from the original PostGIS plan to plain Postgres with a JSON geometry column, since PostGIS required an hours-long from-source GDAL/protobuf/LLVM build on this machine's unsupported OS version and would only have bought a live spatial-query engine the project never actually uses (all spatial computation already happens offline in Milestone 2). See `docs/ARCHITECTURE.md` and `backend/README.md`.

- [x] **Milestone 4 — Frontend scaffold & design system**
  Next.js 16 (App Router, TypeScript, Tailwind v4) initialized in `frontend/`. Dark-first design system (Fraunces + Inter fonts, per-city accent colors, class-based dark mode toggle via `next-themes`), header/footer layout shell, and a typed API client (`lib/api.ts`) verified end-to-end against the live backend — the placeholder homepage renders real numbers (1,652,117 total incidents) fetched server-side from Postgres through FastAPI. Production build, lint, and a Playwright screenshot check (dark + light mode, no console errors) all pass. Public Vercel deployment deferred to Milestone 10, once there's an actual homepage worth showing.

- [x] **Milestone 5 — Homepage hero & scrollytelling intro**
  Full-viewport hero (scroll-linked parallax fade, ambient seeded dot field, per-city accent glows) followed by a four-beat scrollytelling narrative: the project's premise (responsibly reframed from its true-crime-reporting origin), the real data scale (animated count-up of the actual 1,652,117 incidents and per-city breakdown), the data's limits stated up front (not buried — ties directly to `docs/ETHICS.md`), and a closing transition into the sections still to come. Verified with a scripted Playwright scroll-through in dark mode, light mode, and a mobile viewport; caught and fixed a real bug where the count-up animation silently never completed on genuine incremental scrolling (rewritten from Framer Motion's `animate()`/`useMotionValue` to a plain `IntersectionObserver` + `requestAnimationFrame` tween). See `frontend/README.md` for implementation notes.

- [x] **Milestone 6 — Interactive map section**
  A pinned/sticky MapLibre map (free, tokenless OpenFreeMap basemaps, including a real dark style) inside a scroll-driven section: as the reader scrolls, the map flies between Chicago, NYC, and San Francisco, recoloring each city's neighborhoods as a choropleth by incident density in that city's accent color, with a hover tooltip showing exact counts. Required a new backend endpoint (`GET /api/neighborhoods/geojson`) that bundles boundaries + incident totals into one simplified-geometry payload per city (SF: 1.7MB &rarr; 33KB; NYC: 4.4MB &rarr; 198KB). Caught and fixed two real, silent rendering bugs during verification (MapLibre overriding the container's CSS position via inline styles, and a negative-z-index stacking escape) — both only visible by actually reading Playwright screenshots, not from console output. See `frontend/README.md` for details.

- [x] **Milestone 7 — Trend & comparison charts**
  Two custom D3 visualizations, both reading real live data: a monthly time-series line chart (one line per city, hover tooltip with exact values) and a category comparison chart (top 8 categories, shown as each city's own share, not raw counts, so NYC's volume doesn't drown out Chicago/SF). Needed two new backend endpoints (`GET /api/trends/monthly`, `GET /api/trends/categories`). Caught a real data-integrity bug in the process: the current calendar month is always incomplete at pipeline-pull time (SF's partial month sat at ~430 incidents vs. a normal ~7,500-8,000, which would have shown a misleading cliff in the chart) — fixed at the API level by excluding any in-progress month, not just patched in the chart. See `frontend/README.md` for the D3 implementation pattern.

- [x] **Milestone 8 — Case study narrative sections**
  Three case studies, one per city, all found and verified via live web search against official/primary sources (not recalled from memory) and approved by the user before any copy was written: Chicago (ProPublica's 2025 investigation into Illinois's failure to review domestic violence deaths, naming two real individuals via already-published court records), NYC (the city's own Domestic Violence Fatality Review Committee statistic, anonymized), and SF (the city's Domestic Violence Death Review Team pilot report on a 2014 case, anonymized). Full sourcing notes, including one statistic deliberately left out because it couldn't be independently re-verified, are in `docs/CASE_STUDIES.md`. The homepage's closing section was rewritten from a "coming soon" teaser into an actual conclusion, since this was the last planned narrative section.

- [x] **Milestone 9 — Polish pass**
  Concrete, verified fixes rather than a vague pass: (1) **contrast** — programmatically checked every text/background color pair against WCAG AA and found light mode's Chicago/NYC accent colors failed at small text sizes (3.45:1 and 4.51:1 vs the 4.5:1 threshold); darkened both to safe values without touching dark mode (which already passed). (2) **reduced motion** — audited every animated component and found `Reveal` (the most-used one on the site) didn't check `prefers-reduced-motion` at all; fixed it plus the hero's entrance animations, the ambient background's pulsing dots, and the map's fly-to transitions. (3) **responsive** — found the category chart's fixed 168px label margin ate ~43% of a 390px mobile screen; made it width-aware. (4) **performance** — dynamically imported the map (maplibre-gl is ~1.1MB minified) and both D3 charts so their JS doesn't block the initial page load, verified via real bundle size measurement. (5) **cold-start UX** — built `app/loading.tsx` (Next.js's automatic Suspense boundary for the homepage's async data fetch) with a themed "waking up the data" sequence, verified end-to-end with a real (temporary) artificial backend delay, not just code review.

- [x] **Milestone 9.5 — Storytelling/visualization feedback round**
  After using the built site, user feedback was that it still read as "just graphs" and asked for more visualization of scale (e.g. "1 in 30, show the people"), more interactivity, and more urgency/impact. Built isotype-style pictogram visualizations (`components/ui/pictogram-grid.tsx`): Chicago's domestic-flag rate (~4 in 20, required adding `chicago_domestic_flag_count` to `/api/cities`/`/api/summary`) and Illinois's county review-board rate (7 of 102, in the Chicago case study card) — every count traces to an exact, already-sourced figure, never an invented illustration. A separately-raised idea (a 911 call audio moment at the top, for urgency) was not built as proposed: real 911 recordings capture real people's real trauma even when technically public record, and a fabricated "realistic" call would misrepresent a real kind of event — flagged directly rather than built around silently. A follow-up round then: (1) fixed the pictogram icons, which read as unrecognizable dots at their original size once seen live, and (2) built a small persistent pulse (`components/ui/signal-pulse.tsx`) in the hero as the responsible alternative for the urgency beat. That pulse turned out too subtle to register as intentional once seen live ("I don't see the pulsing animation at all," and the site still didn't feel like storytelling) — replaced/supplemented with `components/ui/intro-overlay.tsx`, a full-viewport, skippable title card that plays on every fresh load: three short beats pairing a growing, repeating signal with narrating text ("One contact with police." / "Then another." / "Then a pattern begins.") before fading into the hero. The user then re-raised the 911-call idea directly; given a choice between real/fake audio, generic scripted audio, or an abstract no-audio UI, they chose abstract — added as a fourth opening beat: a voice-memo-style interface (waveform, live duration counter, "911 · Call in progress" pill) with no audio, synthesized voice, or words at all. Also built: click-to-filter charts and map (`lib/filter-context.tsx`) — clicking a category in the category-comparison chart now filters both the monthly trend chart (refetched client-side by category) and the map's choropleth (refetched via the existing per-city-per-category geojson endpoint) together; clicking a city in the (now-interactive) legend isolates its line/bars in both charts. Expandable case study cards remain prioritized but not yet built.

- [x] **Milestone 9.6 — Opening rethink: Chapter 1, "The Call"**
  The user asked for a full rethink of the opening: not a hero banner, but the first 30 seconds of a documentary — a black screen, a dispatch console cold open, a scroll-driven shrink into large typographic statements, and the start of a documentary "chapter" structure (6 planned chapters; only Chapter 1 built so far, per "implement one chapter at a time"). Replaced `hero-section.tsx`, `intro-overlay.tsx`, and `signal-pulse.tsx` entirely with `components/sections/call-chapter.tsx`, built from new reusable dispatch primitives (`components/dispatch/`: badge, organic waveform, live timer, sequenced transcript) and a new cinematic visual layer (`components/ui/scene-atmosphere.tsx`: cursor-spotlight, vignette, film grain — dark-mode only, since the vignette read as a muddy smear once actually checked in light mode). The transcript's illustrative dialogue ("I don't think he's breathing" / "Stay on the line" / "Units are being notified") is the user's own scripted copy, not real or fabricated audio — a lighter-weight case than the earlier audio decision, but still captioned on-screen as reconstructed/illustrative as a safeguard. Hit a genuine, time-consuming bug along the way: `useTransform` outputs bound directly to a scroll-pinned section's `style` prop stopped reaching the painted DOM (confirmed via bisection — the same `MotionValue` read correctly everywhere except the actual rendered opacity), fixed by driving the transition imperatively through `useMotionValueEvent` + `useMotionValue().set()` instead, the same pattern `map-section.tsx` already used successfully. See `frontend/README.md`'s "Chapter 1: The Call" section for the full writeup. A follow-up request asked for a *real* transcript, specifically a case where police didn't respond promptly — raised directly that searching for "a case that fits a failure narrative" conflicts with this project's own case-selection principle (documented cases, chosen for how well-verified they are, not how damning), rather than searching on request. It turned out unnecessary to search at all: the already-sourced SF case study (Milestone 8) is exactly this — a City DVDRT report on "dispatch handling of repeat calls," quoting "multiple responses to the victim's address." The transcript used that real, cited quote first, but the user pushed back again: it read as a paraphrase, not "what was said during the call." Reading the DVDRT PDF directly (not a search summary — required saving it locally and using the `Read` tool after `WebFetch` couldn't parse this PDF's text layer) turned up an actual direct quote from the caller's own 10:10pm call that night (her fourth of six), transcribed verbatim in the report's chronology, including a natural speech filler ("um") confirming it's pulled from the real call record. Fixed the cold-open pacing in the same round (mostly a bare timer before any text appeared) — tightened so the sequence resolves much faster. A third pushback followed: with only one quote, it didn't read as "a conversation" — correctly, since the source contains exactly one verbatim excerpt and inventing a dispatcher's half would repeat the fabrication problem already ruled out. Rebuilt the transcript around four separately-cited real facts instead of one quote alone: how many times she called (six), her real words on one of those calls, how many times police came (three), and that it didn't stop what happened. `case-study-card.tsx` has a stable `id` per city so the citation still links to the full case study. Full chapter system (floating chapter nav, Chapters 2-6: The Response/The City/The Data/The Stories/Explore) and expandable case study cards remain open for future rounds.

- [x] **Milestone 9.7 — Chapter 2, "The Response"**
  Continued the chapter system with a real, cited animated timeline (`components/sections/response-chapter.tsx`) — the same San Francisco case from Chapter 1, laid out minute by minute: six 911 calls, three police visits, an ~4-hour custody hold, and the outcome. Checked `docs/DATA_DICTIONARY.md` before starting and confirmed none of the three cities' datasets include a response-time/dispatch-arrival field, so the brief's "response statistics, delays" can't honestly be built as dataset-wide numbers — flagged this and built the one thing the data *does* support instead: the fully-documented timeline of a single real case, reusing `Reveal` (not a new pinned-scroll mechanism, deliberately, after Chapter 1's `useTransform` bug) for each row's fade-in. Kept a real nuance intact rather than forcing a "slow response" narrative: the source never characterizes any individual visit as slow (each happened within minutes of a call) — the documented failure is a gap in shared context between separate responses, not response speed, and the chapter's copy says so directly. Also caught and fixed a sourcing error carried over from Chapter 1: the 10:10pm call (reused here) was mislabeled as the caller's "third" call — it's her fourth, per the source's own count.

- [x] **Milestone 9.8 — Chapter 3, "The City"**
  Rebranded rather than rebuilt: Milestone 6's pinned interactive map + the Milestone 9.5 click-to-filter integration already delivered most of what this chapter needed (fullscreen map, density-by-neighborhood coloring, hover-based neighborhood exploration), so redoing it from scratch would have duplicated tested work. Changed the overlay's eyebrow to "Chapter 03 · The City" with a one-line intro matching Chapters 1-2's framing, and added a real fade transition (`fill-color-transition`/`fill-outline-color-transition`, 700ms) so a city switch or category filter now visibly fades the choropleth in rather than snapping instantly — the "incidents fade in" beat from the brief. The choropleth itself stands in for "heatmaps" deliberately: the data is pre-aggregated to neighborhood polygons, not raw incident points, so a literal point-density heatmap layer would be the wrong tool for this data's actual shape.

- [x] **Milestone 9.9 — Chapter 4, "The Data"**
  Rebranded the existing Milestone 7 charts rather than rebuilding: they already handled the data honestly (share-of-total not raw counts, excluded the always-incomplete current month), so the only real gap versus the brief was "every chart should animate into view" — both previously just snapped fully-drawn into existence. Added `hooks/use-in-view-once.ts`, a plain `IntersectionObserver` hook (not Framer Motion's `whileInView` — mixing that into a component whose real work is an imperative D3 redraw is exactly the kind of interaction that caused Chapter 1's scroll-linked bug). The line chart now draws itself in via the classic `stroke-dasharray`/`stroke-dashoffset` technique; the category bar chart's bars grow from zero width, staggered per row, with labels fading in just after. Both play only once, ever (a `hasDrawnRef` flag) — later redraws from clicking a filter render instantly rather than replaying the entrance, and both fully respect `prefers-reduced-motion`. Deliberately did not build a new pinned-scroll mechanism for "never show all charts at once" after Chapter 1's bug with that same pattern — the two charts already sit far enough apart in normal scroll flow that only one is ever the one currently animating into view.

- [x] **Milestone 9.10 — Chapter 5, "The Stories"**
  Rebranded and extended the existing Milestone 8 case studies rather than rebuilding. Made cards expandable (collapsed to city/badge/headline by default, click to reveal full content) — a Framer Motion `height: auto` transition, which also closed out the long-standing "expandable case study cards" backlog item from the Milestone 9.5 round. Added a real, dated timeline to Chicago's card built entirely from facts already in its own sourcing (no new claims); SF's card links to Chapter 2's timeline instead of duplicating it; NYC's card deliberately has no timeline, since its source is a citywide statistic, not a documented incident — forcing one would invent structure the source doesn't have. Added neighborhood-level location ("SOMA, San Francisco," from the DVDRT report's own chronology) only where the source supports that granularity, and record-type badges (investigative journalism vs. government report) for all three. Deliberately did not build photographs (no rights-cleared, ethically appropriate images of these real people exist, and a technical licensing check doesn't resolve the dignity/exploitation concern) or a separate incident map per card (Chapter 3 already has the real interactive map; a second, coarser one per card would add weight without information).

- [x] **Milestone 9.11 — Chapter 6, "Explore"**
  Closes out the six-chapter system with a free-form city + category browser (`components/sections/explore-chapter.tsx`), placed just before the closing section. No new backend work — reused the existing `/api/aggregates` (per-neighborhood monthly rows, summed client-side by neighborhood) and `/api/categories` (full ~21-category list, not just the top 8 already charted in Chapter 4) endpoints. Deliberately uses its own local `useState` rather than the shared `FilterProvider` context Chapters 3-4 use, so picking a category here doesn't silently re-filter chapters already scrolled past — a self-contained sandbox, not another connected narrative piece. Surfaces an honest "Unknown" neighborhood bucket for incidents that didn't match a boundary during the Milestone 2 geospatial join, rather than silently dropping them.

  With Chapter 6 built, all six chapters from the original brief exist (The Call, The Response, The City, The Data, The Stories, Explore).

- [x] **Milestone 9.12 — Floating chapter navigator**
  Closes out the last unbuilt piece of the original brief: `components/ui/chapter-nav.tsx`, a "CASE FILE" rail fixed to the left edge (`hidden lg:block` — a persistent side rail doesn't fit narrow screens, and every chapter is still reachable by scrolling), listing all six chapters and highlighting whichever one is current. Uses a plain `IntersectionObserver` "scrollspy" (a thin `-35%/-55%` detection band near the top third of the viewport) rather than a scroll-linked Framer Motion value, deliberately — this only ever needs to know *which* chapter is active, never an exact scroll position, and Chapter 1's `useTransform` bug (Milestone 9.6) is reason enough to keep it simple. The active indicator is a `layoutId`-based sliding bar (Framer Motion's shared-element animation), so it glides between chapters instead of jumping. Stays hidden until the reader scrolls past the opening beat (`window.scrollY > 0.75 * innerHeight`), fades in/out with `AnimatePresence`, and both that fade and the indicator's slide collapse to near-instant under `prefers-reduced-motion`. Added a stable `id="chapter-N"` to every chapter section (renaming Chapter 2's from `chapter-response` to `chapter-2` to match the scheme, and updating `case-study-card.tsx`'s cross-link accordingly) and global reduced-motion-gated `scroll-behavior: smooth` so clicking a nav entry glides rather than snaps. Verified via Playwright: hidden at the very top of Chapter 1, appears on scroll, highlight correctly follows the reader from Chapter 1 into Chapter 2 and all the way to Chapter 5 on a direct nav click, absent on mobile viewport, and present (without the slide/fade animation) under reduced motion.
